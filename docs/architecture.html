Edgebind Oracle Architecture

Human-backed AI task execution, not a marketplace.

The product should guarantee one clean loop: create task, accept task, submit proof, validate proof, decide payout path, then move money only after validation.

Final target: web for owners, mobile for workers, and one shared Next.js backend/runtime. The current web worker route is temporary.

Frontend

Owner Web App

  • Landing page and owner dashboard
  • Create tasks and review execution state
  • Approve only high-risk payouts
  • Deploys on Vercel with the shared API runtime

Shared Runtime

Next.js API + Server Logic

  • Owns task lifecycle, validation, and payout routing
  • Persists users, tasks, submissions, validations, payouts
  • Acts as the single backend for web and mobile
  • Integrates World, Hedera, and Ledger

Mobile

Worker App

  • Browse available tasks
  • Accept assigned work
  • Capture image and location proof
  • Submit proof and view payout outcome

Core Execution Flow

1. Owner

Human-backed agent owner creates a task with reward, deadline, proof type, and optional location.

2. World

Verified human worker identity gates participation. World also anchors the human-backed agent story.

3. Worker

Worker browses and accepts the task from mobile.

4. Proof

Worker submits image, location, time, and request code.

5. Validation

Backend checks deadline, assignment, proof fields, and location constraints.

6. Decision

Agent decision chooses auto pay or requires approval.

7. Money

Hedera pays low-risk tasks. Ledger appears only for high-risk approval before payout.

Integrations

Must-Have Stack

Roles of Each Integration

World = identity layer

Hedera = payout rail

Ledger = approval path only

0G = deferred

  • World must gate verified workers and human-backed agent trust.
  • Hedera must be the payout mechanism for successful execution.
  • Ledger must only appear when a payout requires manual approval.
  • 0G should not block the core demo and stays out for now.

Current Reality

Where The Repo Is Today

  • Next.js runtime and Postgres-backed task lifecycle already exist.
  • Owner web flow exists and is the right long-term surface.
  • Mobile worker app exists but still speaks the old request/verify API.
  • Temporary web worker flow exists only as a fallback test surface.
  • World, Hedera, and Ledger are still pending integration.

Architecture Cleanup Path

What Stays

Keep

  • `frontend` as the only backend/runtime on Vercel
  • `frontend/owner` as the real owner experience
  • Postgres as the persistent execution core
  • `mobile` as the long-term worker client

What Changes

Converge

  • Align mobile from `requests/verify` to the shared `tasks` lifecycle
  • Move mobile creation UX out of the worker app
  • Reduce or remove the temporary web worker surface after alignment
  • Add World first, Hedera second, Ledger third
