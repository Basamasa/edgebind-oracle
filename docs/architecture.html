Developer Architecture Graph

Edgebind Oracle should be one execution system with one owner app, one worker app, and one shared backend.

This repo should not feel like two separate products. The clean target is: web for owners, mobile for workers, and Next.js as the only backend/runtime.

Current problem: the repo still has a temporary web worker flow while the mobile app speaks an older API. Final direction: converge everything on one task lifecycle and remove duplicated worker product surfaces.

2. Core Execution Loop

The product is not a marketplace. It is a proof-and-payment execution layer.

1. Create

Owner, acting through a human-backed agent, creates a task with reward, deadline, proof type, and location rules.

2. Gate

World should verify who can participate: trusted owner identity and verified worker identity.

3. Accept

Worker accepts the task. This must be a real lifecycle step, not implied by proof submission.

4. Submit

Worker sends proof payload: image, coordinates, timestamp, request code, and task reference.

5. Validate

Backend validates the proof against assignment, deadline, proof type, and location constraints.

6. Decide

Agent decision chooses auto_pay or requires_approval.

7. Pay

Low-risk tasks pay through Hedera. High-risk tasks route to Ledger approval first, then pay.

5. Integration Placement

Each external system should have one clear purpose.

World

Identity Gate

Must gate participation for verified workers and support the human-backed agent story for owners.

worker verification owner trust

Hedera

Payout Rail

Must be the payment path after successful validation and decisioning. It should not appear before validation.

auto pay receipt / tx reference

Ledger

Manual Approval

Must only appear for high-risk payouts. It should not be in the normal low-risk happy path.

approval only high risk branch

6. Current Gap To Close

The main architectural gap is contract mismatch, not missing screens.

Mobile Today

Old Contract

  • GET /api/requests
  • POST /api/verify
  • request/proof language
  • implicit one-step verification flow

Backend Today

New Contract

  • GET /api/tasks
  • POST /api/tasks/:id/accept
  • POST /api/tasks/:id/submissions
  • task/accept/submit/validate/pay language

7. Recommended Convergence Path

Phase A

Keep The Core

Keep Next.js as the only backend/runtime and keep the owner web surface.

Phase B

Align Mobile

Move mobile from requests/verify to the shared task lifecycle, either directly or through short-term compatibility endpoints.

Phase C

Remove Duplication

Once mobile is aligned, de-emphasize or remove the temporary web worker route and keep one real worker client.

Summary: the architecture becomes open-source-worthy once the repo clearly communicates one owner web app, one worker mobile app, one shared Next.js backend, and one task lifecycle that all clients follow.
